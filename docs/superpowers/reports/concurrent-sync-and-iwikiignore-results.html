Требования по разделам

РазделТребование
§ProblemЗафиксировать два пробела при конкурентном доступе: (1) гонки git на общей base (git add -A захватывает чужие файлы; коллизии index.lock; коллизии push non-fast-forward; неатомарность write→log→reindex между процессами); (2) отсутствие фильтра контента (.iwikiignore — стаб, не задействован).
§Scope & usage modelПараллельные проекты пишут в разные write-домены в одной общей base. Содержимое файлов не сталкивается; сталкиваются только общий git-индекс и push. На блокировке — ждать с таймаутом + retry push на non-fast-forward. Межпроцессная блокировка через библиотеку filelock. .iwikiignore держит секреты/шум вне вики, seed из .gitignore, enforced server-side как gate на аргумент source= в wiki_write_page.
§Architecture overviewДве независимые fail-soft фичи. Concurrency: межпроцессный lock на base + commit, ограниченный pathspec write-домена + push-retry. .iwikiignore: gitignore-файл в корне проекта; gate на source= при write + автосоздание при init домена / bind проекта. Новые модули: lock.py, ignore.py. Правки: sync.py, server.py, pyproject.toml.
§Concurrency design / lock.pybase_lock(base, timeout=15.0) → FileLock; lock-файл base/.iwiki/lock; base/.iwiki/ — метаданные base-уровня, не домен, не стейджится; на таймаут — filelock.Timeout, транслируется в fail-soft warning.
§Concurrency design / sync.pyauto_commit(base, message, pathspec=None, timeout=15.0) — lock вокруг add+commit; git add -- <pathspec> (директория write-домена); pathspec=None → прежнее поведение; форма возврата неизменна; Timeout → {committed:False, warning:"base busy: lock timeout"}. sync(base, timeout=15.0, push_retries=3) — lock вокруг pull/push; pull --rebase → push; на non-fast-forward retry до push_retries раз; конфликт rebase → rebase --abort + прежний error/hint; Timeout → warning. Тайминги — аргументы функций с дефолтами, не через Config.load.
§Concurrency design / server.py wiringwiki_write_page и wiki_create_domain вызывают sync.auto_commit(bind.base, msg, pathspec=valid_domain).
§.iwikiignore / ignore.pyensure_iwikiignore(project_dir)→bool (идемпотентно, True если создан); load_project_ignore(project_dir)→PathSpec|None (компиляция .iwikiignore, None если нет/нет реальных паттернов); is_ignored(spec, source, project_dir)→bool (внутри project_dir → relpath match; снаружи → basename match; spec None → False).
§.iwikiignore / EnforcementВ wiki_write_page до начала транзакции, если задан source: spec=load_project_ignore(bind.project_dir); если is_ignored → возврат {error:"source matches .iwikiignore", hint:…}; ничего не пишется/логируется/индексируется/коммитится (gate до транзакции, без rollback). source опционален → gate срабатывает только если путь задан (defense-in-depth, не абсолютная гарантия).
§.iwikiignore / Creationensure_iwikiignore(bind.project_dir) вызывается из wiki_create_domain и wiki_bind; оба идемпотентны. Seed из .gitignore — однократная копия при создании; в runtime читается только .iwikiignore.
§Error handlingВсё через @_safe envelope; lock timeout, push failure, ignore-match → warning/error dict, никогда не raise; gate отклоняет до транзакции (без rollback); существующая транзакционная гарантия write→log→index неизменна.
§Testingtests/test_sync_concurrency.py (pathspec-scoped commit; base_lock блокирует/освобождает; push-retry сходится) и tests/test_iwikiignore.py (ensure создаёт/не перезаписывает/seed; write отклоняет при совпадении source; create_domain и bind создают .iwikiignore). Паттерн: monkeypatch indexer.embed_texts, dummy IWIKI_* env; filelock реальный.
§DependenciesДобавить filelock в runtime-зависимости pyproject.toml; pathspec уже зависимость.
§Affected filesEdit: sync.py, server.py, pyproject.toml; New: lock.py, ignore.py, test_sync_concurrency.py, test_iwikiignore.py; Docs: обновить indexing.md и/или architecture.md.
§Out of scopeИндексирование исходников проекта напрямую; удаление уже проиндексированных страниц через ignore-паттерны; per-domain .iwikiignore внутри base; конкурентная запись в один домен из двух процессов (write→log→reindex не залочен).

F-001 WARNING ## Problem write file → append log → reindex is not atomic across processes. Симптом #4 (неатомарность write→log→reindex между процессами) указан в Problem как пробел гонок git, но Concurrency design оборачивает base_lock только auto_commit (add+commit) и sync (pull/push); последовательность write→log→reindex в wiki_write_page выполняется до auto_commit и блокировкой не покрыта. Модель использования (разные write-домены → изолированные по доменам index-файлы) неявно нейтрализует это, но спека нигде не утверждает, что симптом тем самым решён или вне области. Добавить одно предложение (в Concurrency design или Out of scope), явно отметив, что симптом #4 нейтрализован изоляцией по доменам (каждый проект пишет свой <domain>/.iwiki/index.jsonl) или иным образом покрыт. Внесено: §Concurrency design и §Out of scope теперь явно отмечают нейтрализацию изоляцией по доменам и same-domain-конкуренцию вне области. fixed (2026-06-30)
