iwiki-mcp — How the Server Works

A stdio MCP server that fronts a shared, git-synced wiki base split into domains. Coding agents author Markdown pages; the server validates structure, persists, embeds, indexes, and runs hybrid (vector + lexical) search across the domains a project is bound to.

sources: src/iwiki_mcp/** · docs/wiki/*.md · generated 2026-07-01 · branch master

Not a daemon. The server lives for the client session that spawned it. iwiki-mcp = iwiki_mcp.server:main parses an optional --project DIR, then mcp.run() speaks MCP over stdio via a FastMCP("iwiki") instance.

1. What the server is

Two layers
Top layer is MCP-aware (server.py, base.py, indexer.py, retrieval.py, sync.py, lock.py, ignore.py, resources.py). The engine/ core is framework-free and unit-testable.
On-disk model
A base dir is a git repo; each immediate subdir is a domain holding *.md pages plus .iwiki/index.jsonl (embedding store) and .iwiki/log.jsonl (ingest log).
Binding
A project's .iwiki.toml binds read = [...] domains and one write domain. base.resolve_binding() merges it with IWIKI_* env vars.
Fail-soft surface
Every wiki_* handler is wrapped by @_safe: it never raises — exceptions become {"error","hint"} dicts, so the MCP client always gets JSON.

6. Load-bearing invariants

Fail-soft. Every wiki_* is wrapped by @_safe. Implementations are plain functions registered separately (mcp.tool()(wiki_*)) so tests call them directly — keep this split when adding tools.
HALT stop rule. Config.load() raises ConfigError if IWIKI_LLM_BASE_URL/IWIKI_LLM_KEY are unset; a persistent embedding failure raises EmbedError. Both surface through @_safe as HALT: errors. Missing base/binding raises base.BaseError.
Path-traversal guards are load-bearing. _validate_domain, _slug_parts, and _contains reject .-prefixed, absolute, drive-letter, and .. paths before any filesystem join.
Chunking & indexing details (click to expand)
  • Split on ## only. The first ## Overview section is the article summary and is excluded from the index.
  • Context-carrying vectors. Every other section's sub-chunks are prefixed with # Title + article summary + ## heading + section lead (≤ LEAD_MAX 250), so each vector carries whole-article + whole-section context.
  • Incremental. A chunk is reused when its hash and dim match the stored record; only changed chunks hit the embeddings API.
  • Cap. index_domain flags over_cap past 8 MB (CAP_BYTES).
  • Duplicated constants. OVERVIEW_HEADING / LEAD_MAX / the _H2 regex are copied across chunk.py, validate.py, lint.py on purpose — lint/validate must stay stdlib-only (no httpx). Change one copy → change all.
Git & concurrency details (click to expand)
  • Best-effort git. A non-repo, missing remote, or rebase conflict degrades to a warning/error dict — never an exception.
  • Domain-scoped commits. auto_commit stages only the written domain's pathspec.
  • Cross-process lock. Many servers (one per client session) can share one base; base_lock (a filelock at <base>/.iwiki/lock) serializes all git mutations.
  • Push retry. wiki_sync retries a non-fast-forward push up to 3 times; a pull --rebase conflict is aborted cleanly with a resolve hint.
