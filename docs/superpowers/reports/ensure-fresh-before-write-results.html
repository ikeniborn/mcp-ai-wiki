1. Резюме спецификации

Файл: docs/superpowers/specs/2026-07-01-ensure-fresh-before-write-design.md · spec_hash: e03bfe0bcdeec208 · вердикт: OK

Цель: тянуть remote до локальной мутации, чтобы запись ложилась на текущий remote-tip и push становился fast-forward. При реальном расхождении (diverged) — отказать в записи с hint, не громоздя новый коммит на расхождение.

Требования по секциям

IDСекцияТребование
R1Design §1Новая fail-soft функция sync.ensure_fresh(base): fetch + при чистом отставании merge --ff-only; никогда не бросает; берёт base_lock.
R2Design §1ahead/behind считать через git rev-list --left-right --count @{upstream}...HEAD.
R3Design §1Модель состояний: no_repo/no_remote/no_upstream/offline/up_to_date/ahead/updated/dirty/diverged с заданной реакцией.
R4Design §2Вызов ensure_fresh в начале всех 5 mutating-хендлеров (write/update/delete/reindex/create_domain) до validate/write.
R5Design §2diverged → немедленный возврат {error, hint} с нулём сайд-эффектов (нет .md/лога/индекса).
R6Design §2Любой warning (offline/dirty/no_upstream) прокидывается в итоговый result рядом с committed/pushed. wiki_sync не трогаем.
R7Design §3Тесты: unit по состояниям ensure_fresh + хендлер-уровень (diverged→refuse zero-effect; behind→ff+write+push). Паттерн no-network.
R8Design §4Docs: wiki sync-страница, README (только если описывает git-поведение записи), project CLAUDE.md; patch-bump 0.1.4→0.1.5; строка в docs/TODO.md.
